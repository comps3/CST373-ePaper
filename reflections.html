This issue comes down to the level of severity in the consequences a user faces due to faulty software. If Chrome (Internet browser) decided to crash just before an assignment was due, I would be tempted to blame the developers but I would also take responsibility for turning in an assignment at the last minute. In ten years when self-driving vehicles become more reliable and in the scenario where I were to be involved in an accident; I would blame the software developers of the vehicle for any injury I would have. Although, I would be sympathetic towards the Software Engineers because it is difficult to write software that can anticipate every possible scenario that can occur in life. Software development tools and methodology will undoubtedly improve as time goes on which will lead to a decrease in software defects.

After writing this paper, I have gained a greater understanding about an issue that plagues Software Engineers. At the start of this paper, I believed that Software defects were inevitable and Software Engineers are responsible to write good code and perform extensive testing before a product was shipped. However, NASA has already shown that it requires a large amount of resources in order to create near perfect software. In the private industry, software may be unable to be run through the test created by the quality assurance engineers due to the timeline requirement. Luckily, the media has devoted portions of their time to report on serious Software bugs that have the potential to impact people’s lives.

Before writing this paper, I did not actually really think that software engineers should be blamed for any accidents that happen due to bugs. Perhaps, I was bias sense that is the career I chose. After analyzing different incidents that have happened throughout the years, it became more clear to me why software engineers do not have absolute control of the consequences of the code.

In some cases, it is not even that the software developers did not do their job when creating the code, but the fact that the requirements or variables of the situation change. Overall, my opinion in this issue did not change. I still believe that software engineers should no be blamed for incidents that happen as a consequence of bugs.

At the moment, when users have complaints about their software experience they take to their favorite social media platform to discuss their concerns. Instead, they could that the same amount of time to reach out to the company that sold them the software and hold a conversation about how the software defect was triggered. This concept does seem quite similar to the widely regarded technical support experience. Due to the availability of high quality search engines, users will be able to resolve some technical problems without the company’s assistance. Users will be able to report problems related to the inner workings of the program. The incentive for users to participate in such program is due to the inherent reciprocation of good. Users have the ability to improve the software experience for all the people who use it.

In the political realm, there are no government entities that oversee problems in the Software industry. As Software continues to consume more aspects of our lives, the government should create an entity similar to the National Highway Traffic System Administration. Its initial duties could be to hold companies responsible for the damages that users experience through use of their software.

Before writing the paper and conducting research on our topic, I believed that software engineers were the ones to be mostly blamed if an accident happened. The other part of the blame would go to the user for fully trusting a technological device or a certain system. Users should expect that code is not created perfectly therefore at some point a bug will be found. As a software engineer, I can say that there are different tests that a code must pass, but it is hard to achieve passing the tests plus showing its correct functionality.

I have seen different situations were a code can break and output a different result than what is expected. It is a normal process for engineers to fix their mistakes after releasing a product due to the fact that the Quality Assurance Engineers cannot provide all of the possible scenarios a code can go through. The error can occur because of a software bug or the software is being used in a device that was not tested by the engineers.

After doing research and reading about various cases of software bugs, I now have a better understanding of why there are software bugs in a software and why perfect code cannot be released. In order to release a close to perfect code and well maintained code, a lot of resources are needed, such as money, time, and high software development skills. The only company that has been able to reach a release of perfect code has been NASA, which have spent around one-thousand dollars on every line of code in order to maintain it. Time also plays a big part of bug free code, engineers can have short deadlines therefore they have to finish the faster than usual. This can cause a lot of bugs to appear because of its rapid development.

Some recommendations for the future social changes is that people test the new software carefully before fully trusting and relying on it. Another option can be that only a certain amount of people can use the software and test it to see what kind of bugs are encountered. More time should be given to the engineers to develop the software code and not have them feel pressured to meet short deadlines.
